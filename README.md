## 2. Introduction & System Purpose

The **GymOps** system is designed to streamline and automate the day-to-day operations of a modern fitness center. Managing a gym requires tracking a complex, dynamic set of relationships between members, staff, physical assets, and financial transactions. 

**Key Data Stored in the System:**
* **Members & Subscriptions:** Personal information, membership tiers, subscription start/end dates, and payment statuses (Active, Frozen, Expired).
* **Equipment & Inventory:** Detailed logs of gym machinery (treadmills, weights, machines), their current operational status, and maintenance schedules to minimize downtime.
* **Staff & Operations:** Employee details, roles (e.g., trainers, maintenance, front desk), and activity logs.

**Main Functionalities:**
Our primary goal is to perform efficient CRUD (Create, Read, Update, Delete) operations that allow management to monitor active memberships, quickly identify equipment requiring repair, and analyze facility usage. By centralizing this data, GymOps provides an operational overview that improves resource allocation and enhances the customer experience.

## 4. Database Design: ERD & DSD

Thorough database modeling was conducted prior to implementation to ensure data integrity and query efficiency. The design files are included in the repository: `GymOps.erdplus` and `GymOps-Relational Schema.erdplus`.

### Entity Relationship Diagram (ERD)
The ERD maps out the conceptual structure of the GymOps system, highlighting the cardinalities between users, equipment, and subscriptions.
![GymOps ERD](DBProject_216680231_216523787/images/Programing/image3.jpeg)

### Data Structure Diagram (DSD) / Relational Schema
The DSD translates our conceptual model into a strict relational schema, detailing primary keys (PK), foreign keys (FK), and data types.
*(Note: If you have a specific image for the DSD, replace the path below)*
![GymOps DSD](DBProject_216680231_216523787/images/DataImportFiles/image2.jpeg)

### Design Decisions & Justifications
1.  **Normalization (3NF):** We strictly adhered to the Third Normal Form (3NF) to eliminate data redundancy and prevent insertion, update, and deletion anomalies. For instance, membership tier details are stored in a separate table rather than being duplicated in every user record.
2.  **Separation of Concerns:** Equipment inventory is decoupled from the maintenance logs. This allows us to keep a historical archive of repairs for a specific machine without cluttering the main equipment lookup table.
3.  **Junction Tables for M:N Relationships:** Where necessary, we implemented associative entities (bridge tables) to resolve many-to-many relationships, ensuring the database remains scalable and queries perform optimally.
